강력 추천

한 줄 결론: Claude Code 스킬만으로 실시간 BTC 트레이딩 봇을 돌리는 것은 택시를 타고 F1 레이스에 나가는 것과 같습니다. 택시(Claude)는 훌륭한 이동수단이지만, 레이스카(TraderJ 엔진)가 이미 있을 때 굳이 택시로 레이스할 이유가 없습니다.

2. 세 가지 축 개별 검토

축 1: TradingAgents 아키텍처를 Claude 스킬로 구현

TradingAgents 컴포넌트 스킬로 구현 가능? 이유
4종 Analyst (Fundamental, Sentiment, News, Technical) 가능 각 Analyst는 독립 프롬프트 → LLM 호출. 스킬 내 서브에이전트로 병렬 실행 가능
Bull/Bear Debate 가능 기존 talk-about 스킬 패턴과 유사. 멀티에이전트 토론 구조 재활용 가능
Risk Debate (3인 토론) 가능 Aggressive/Conservative/Neutral 관점 프롬프트 → 중위값 산출
Reflection Memory 부분 가능 스킬 세션 간 상태 공유 불가. 파일 기반 메모리(JSON/SQLite)로 우회는 가능하나 벡터 검색은 불가
Signal Processing 가능 LLM 출력에서 BUY/SELL/HOLD 추출은 프롬프트 엔지니어링으로 해결
LangGraph 오케스트레이션 부분 가능 스킬 내 서브에이전트 체이닝으로 유사 구현 가능하나, 상태 그래프의 조건부 분기는 제한적
지속적 학습 루프 불가 스킬은 세션 단위 실행. 세션 간 학습 상태 전파 메커니즘 없음

축 2: Schedule 스킬로 무한 루프 트레이딩

Schedule 스킬의 실체

Schedule 스킬은 create_scheduled_task 도구를 호출하여 cron/interval 기반 반복 작업을 등록합니다. 각 실행은 완전히 독립된 새 Claude 세션으로 시작됩니다.

항목Schedule 스킬TraderJ 엔진
최소 주기 cron 기반 (최소 1분, 실질적으로 5~15분 권장) 60초 (APScheduler IntervalTrigger)
실행 지연 세션 시작 10~30초 + LLM 추론 5~30초 = 15~60초 tick() 함수 호출 <1초 (지표 계산만)
상태 유지 불가 (매 실행이 새 세션) 메모리 + DB 영구 보존
비용/실행 ~3,000~10,000 토큰 (세션 부트스트랩 포함) 0원 (로컬 Python)
월 비용 (1분 주기) $200~600/월 $0 (서버 전기료만)
신뢰성 LLM 환각, 타임아웃, API 장애 위험 결정론적, 예측 가능
포지션 추적 매번 DB에서 읽어야 함 (외부 저장소 필요) 메모리 + EventBus 실시간 추적
SL/TP 모니터링 불가 (세션 간 실시간 감시 불가) MarketTickEvent로 매 틱 체크

핵심 문제: "빈 틱" 문제

비유로 설명하면, TraderJ 엔진은 의사가 환자 옆에 24시간 대기하는 것이고, Schedule 스킬은 1분마다 의사를 새로 불러서 처음부터 환자 차트를 읽게 하는 것입니다.

Schedule 기반 실행 (매 1분): ──────────────────────────────────────────────────── [00:00] 새 Claude 세션 시작 (10~30초 부팅) [00:20] 스킬 로딩, 컨텍스트 파악 [00:30] Upbit API 호출 → OHLCV 데이터 수집 [00:35] 지표 계산... 근데 잠깐, 이전 틱의 상태가 없음! - 현재 포지션? → DB에서 읽어야 함 - 미실현 손익? → 다시 계산해야 함 - SL/TP 히트 여부? → 지난 1분간 모니터링 안 됨! ⚠️ - 레짐 디바운스 카운트? → 어딘가에 저장해야 함 [00:50] 시그널 생성 → HOLD (대부분의 틱은 HOLD) [00:55] 결과: "아무 것도 안 함" ← $0.02 소비 ──────────────────────────────────────────────────── 24시간 × 60틱 = 1,440회/일 × $0.02 = $28.80/일 = $864/월 그 중 실제 매매는 하루 2~5회뿐 ← 99.7%가 "헛수고" TraderJ 엔진 (상시 구동): ──────────────────────────────────────────────────── [asyncio event loop - 항상 실행 중] [00:00] tick() 호출 → 0.1초 만에 지표 계산 완료 [00:00.1] 메모리에서 즉시 포지션/상태 접근 [00:00.2] 시그널: HOLD → 0.2초에 완료, 다음 틱 대기 ──────────────────────────────────────────────────── 비용: $0 (로컬 실행)

3. 왜 순수 스킬 기반은 안 되는가 (상세)

구조적 불일치 5가지

1. 세션 격리 (Session Isolation)

Claude 스킬/스케줄은 매 실행이 완전히 새로운 세션입니다. 비유하면 매번 기억을 잃는 의사에게 환자를 맡기는 것입니다. TraderJ의 EventBus, StateMachine, PositionManager는 모두 메모리 상주하며 실시간으로 상태를 추적하는데, 스킬은 이 연속성을 가질 수 없습니다.

2. SL/TP 실시간 모니터링 불가

TraderJ는 매 MarketTickEvent(60초)마다 현재가를 체크하여 Stop Loss, Take Profit, Trailing Stop을 감시합니다. 스킬 기반은 세션이 켜져 있는 순간에만 이런 체크가 가능합니다. 세션 간 1분의 공백 동안 급등/급락이 발생하면 SL이 발동되지 않습니다.

위험 시나리오:
  10:00:00 - Schedule 세션 실행 → BTC = 50,000,000원, SL = 48,500,000원 → "정상"
  10:00:55 - 세션 종료
  10:00:57 - BTC 급락 → 47,000,000원!  ← SL 발동해야 하지만 아무도 감시하지 않음
  10:01:00 - 새 세션 시작...  ← 이미 SL 가격을 뚫고 내려감
  10:01:30 - 세션 부팅 완료, 상황 파악 → "어? SL 뚫렸네" → 시장가 청산
  ────────────────────────────────────────
  결과: 3% 손절이 아니라 6% 손실 (SL 미발동으로 2배 손실)

3. 비결정론적 실행

LLM은 같은 입력에 다른 출력을 낼 수 있습니다. TraderJ의 SignalGenerator는 같은 OHLCV 데이터에 항상 같은 시그널을 생성하지만, LLM 기반은 temperature, 컨텍스트 길이, 모델 버전에 따라 결과가 달라집니다. 이는 백테스트 불가능을 의미합니다.

4. 지연 시간 (Latency)

암호화폐 시장에서 BTC가 1분에 1~3% 움직이는 경우가 있습니다. 스킬 세션 시작(10~30초) + LLM 추론(5~30초) = 15~60초 지연은 슬리피지를 크게 키웁니다.

5. 비용 효율성

60초 주기 실행 시 하루 1,440회, 월 43,200회의 LLM 세션이 필요합니다. 99% 이상이 "HOLD" (아무 행동 안 함)인데도 매번 비용이 발생합니다.

5.2 LLM 클라이언트 설계

# engine/llm/client.py

class LLMClient:
    """
    TraderJ 엔진 내부에서 Claude API를 직접 호출.
    Claude Code 스킬이 아닌, httpx로 Anthropic API를 직접 사용.
    """

    def __init__(self):
        self._client = anthropic.AsyncAnthropic(
            api_key=os.environ["ANTHROPIC_API_KEY"]
        )
        # 비용 절약: 빈도별 모델 분리
        self._models = {
            "fast": "claude-haiku-4-5-20251001",     # 감성분석, 뉴스 스캔
            "deep": "claude-sonnet-4-6",              # Bull/Bear 토론, 회고
        }

    async def analyze(self, prompt: str, tier: str = "fast") -> LLMResponse:
        response = await self._client.messages.create(
            model=self._models[tier],
            max_tokens=1024,
            messages=[{"role": "user", "content": prompt}],
            temperature=0.3,  # 일관성을 위해 낮은 temperature
        )
        return self._parse_response(response)

5.3 텔레그램 양방향 제어 확장

# engine/notification/telegram.py 확장

class TelegramCommandReceiver:
    """
    Telegram Bot의 getUpdates (Long Polling)로 사용자 명령 수신.
    EventBus를 통해 TradingLoop에 명령 전달.
    """

    COMMANDS = {
        "/status":    "현재 봇 상태 + 포지션 + 미실현 PnL",
        "/pause":     "트레이딩 일시정지",
        "/resume":    "트레이딩 재개",
        "/stop":      "봇 정지",
        "/emergency": "긴급 포지션 청산 + 봇 정지",
        "/force_sell":"현재 포지션 시장가 매도",
        "/lock":      "레짐 자동전환 잠금",
        "/unlock":    "레짐 자동전환 해제",
        "/report":    "일일/주간 리포트 즉시 발송",
        "/regime":    "현재 레짐 + 디바운스 상태 조회",
        "/risk":      "리스크 상태 (연속손실, 일일PnL, 쿨다운)",
    }

    async def _poll_loop(self):
        """Long polling으로 텔레그램 명령 수신"""
        while self._running:
            updates = await self._get_updates(timeout=30)
            for update in updates:
                cmd = self._parse_command(update)
                if cmd:
                    # IPC와 동일한 경로로 명령 전달
                    await self._event_bus.publish(
                        BotCommandEvent(command=cmd.action, strategy_id=cmd.strategy_id)
                    )
                    # 실행 결과 텔레그램으로 회신
                    await self._notifier.send(f"명령 실행: {cmd.action}")

질문 답변
Claude 스킬로 TradingAgents 구현? 개별 에이전트는 가능, 전체 오케스트레이션은 불가
Schedule 스킬로 무한 루프 트레이딩? 비추천 — 비용, 안전성, 지연 모두 불리
텔레그램 보고 및 제어? 완전 가능 — 기존 알림은 구현됨, 명령 수신만 추가
최적 전략 하이브리드 — TraderJ 엔진 + Claude API 직접호출 + 텔레그램 양방향

비유로 정리하면: TraderJ 엔진은 심장(항상 뛰어야 함)이고, Claude LLM은 뇌의 고차원 사고(필요할 때 깊이 생각)이며, 텔레그램은 신경계(감각 입력 + 운동 명령)입니다. 심장을 뇌로 대체할 수는 없지만, 뇌가 심장의 박동을 조절하는 것은 자연스럽습니다.
