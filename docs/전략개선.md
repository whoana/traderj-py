# 레짐 및 전략 튜닝 AI 모드 검토

현재 고정된 조건이나 파라미터 값으로 레짐 및 전략을 수행하는 형태에서, 의미 있는 기간 동안 수행한 거래를 AI가 평가하여 레짐과 전략의 파라미터를 수정하고 엔진에 반영하도록 한다.

> **비유:** 지금 시스템은 "매뉴얼 기어차"와 같다. 운전자(개발자)가 직접 기어(파라미터)를 조절해야 한다. AI 튜닝 모드는 이것을 "자동 기어차"로 바꾸는 것이다 — 도로 상황(시장)에 따라 기어가 스스로 최적 위치를 찾는다.

---

## 1. 현황: 현재 파라미터 관리 방식

현재는 grid search로 고정 파라미터 셋을 찾아 `presets.py`에 하드코딩하는 방식이다 (최종 튜닝: 2026-03-07, 90일 walk-forward). 시장 환경이 변해도 파라미터 값 자체는 변하지 않고, 레짐 감지에 따라 프리셋 간 전환만 수행한다.

**문제점:**

- 2026-03-07 튜닝 시점의 시장 특성에 과적합(overfitting)될 위험
- BTC/KRW 시장의 변동성 구조가 바뀌면 기존 임계값이 무효화
- 페이퍼 트레이딩 결과(3/10~3/16)에서 이미 SL 미설정, 매도 미작동 등 이슈 관측

---

## 2. 변경 대상 파라미터

AI 튜닝 모드가 조정할 수 있는 파라미터를 **3단계 계층**으로 구분한다.

### 2.1 Tier 1 — 시그널 파라미터 (고빈도 조정)

매 평가 주기(예: 7일)마다 조정 가능한 핵심 파라미터.

| 파라미터 | 현재 값 범위 | 조정 범위 | 설명 |
|---------|------------|----------|------|
| `buy_threshold` | 0.05 ~ 0.18 | 0.03 ~ 0.25 | 매수 시그널 발동 임계값 |
| `sell_threshold` | -0.03 ~ -0.12 | -0.20 ~ -0.02 | 매도 시그널 발동 임계값 |
| `tf_weights` | {1h: 0.2~0.5, 4h: 0.3~0.5, 1d: 0.3~0.7} | 합계 1.0, 각 0.1~0.8 | 타임프레임별 가중치 |
| `macro_weight` | 0.0 ~ 0.20 | 0.0 ~ 0.30 | 매크로 시그널 반영 비중 |
| `score_weights` (w1, w2, w3) | TF: (0.50, 0.30, 0.20), HY: (0.40, 0.40, 0.20) | 합계 1.0, 각 0.10~0.60 | 서브스코어(추세/모멘텀/거래량) 가중치 |

> **비유:** 이 파라미터들은 라디오의 이퀄라이저 슬라이더와 같다. 저음(추세)·중음(모멘텀)·고음(거래량)의 비율을 조절하여 현재 "음악"(시장)에 가장 맞는 소리를 만든다.

### 2.2 Tier 2 — 리스크 파라미터 (중빈도 조정)

2~4주 단위로 평가 후 조정. 손익에 직접 영향을 주므로 보수적으로 변경한다.

| 파라미터 | 현재 값 | 조정 범위 | 설명 |
|---------|--------|----------|------|
| `atr_multiplier` (SL) | 2.0 | 1.5 ~ 3.0 | ATR 기반 손절 거리 배수 |
| `reward_risk_ratio` (TP) | 2.0 | 1.5 ~ 4.0 | 보상/위험 비율 |
| `trailing_activation_pct` | 0.01 (1%) | 0.005 ~ 0.03 | 트레일링 스톱 활성화 기준 |
| `trailing_distance_pct` | 0.015 (1.5%) | 0.008 ~ 0.03 | 트레일링 스톱 추적 거리 |
| `max_position_pct` | 0.20 (20%) | 0.05 ~ 0.30 | 최대 포지션 비중 |
| `volatility_cap_pct` | 0.08 (8%) | 0.05 ~ 0.12 | 변동성 초과 시 진입 차단 |
| `daily_max_loss_pct` | 0.05 (5%) | 0.03 ~ 0.08 | 일일 최대 손실 한도 |

### 2.3 Tier 3 — 레짐 파라미터 (저빈도 조정)

월 1회 이하 평가. 레짐 감지 체계 자체를 변경하므로 가장 신중하게 접근한다.

| 파라미터 | 현재 값 | 조정 범위 | 설명 |
|---------|--------|----------|------|
| `adx_threshold` | 25.0 | 20.0 ~ 35.0 | 추세/횡보 판단 기준 |
| `bb_width_threshold` | 0.04 | 0.02 ~ 0.08 | 고변동/저변동 판단 기준 |
| `debounce_count` | 3 | 2 ~ 5 | 레짐 전환 확인 횟수 |
| `cooldown_minutes` | 60 | 30 ~ 120 | 전환 간 최소 대기 시간 |
| `regime → preset 매핑` | 6개 고정 매핑 | 프리셋 재배치 가능 | 어떤 레짐에 어떤 전략을 배정할지 |

> **비유:** Tier 1은 자동차의 가속/브레이크 미세 조정, Tier 2는 서스펜션과 타이어 공기압 조정, Tier 3은 엔진 세팅 변경에 해당한다. 위로 갈수록 빈번하게, 아래로 갈수록 신중하게 변경한다.

## 3. AI 튜닝 아키텍처 (하이브리드)

> **핵심 원칙:** 숫자 계산은 통계 엔진(Optuna)이, 맥락 해석과 의사결정은 LLM이 담당한다. 마치 병원에서 MRI 장비(통계)가 영상을 찍고, 의사(LLM)가 진단과 처방을 내리는 것과 같은 역할 분담이다.

### 3.2 AI 프로바이더 선택 및 우선순위

#### 3.2.1 프로바이더 구성

하이브리드 시스템은 **통계 엔진**과 **LLM 프로바이더** 두 축으로 동작한다. 통계 엔진은 항상 로컬에서 실행되고, LLM은 외부 API를 호출한다.

```python
# engine/tuner/provider_config.py (예시 구조)

@dataclass
class LLMProviderConfig:
    name: str                    # "claude", "openai", "local-only"
    model: str                   # "claude-sonnet-4-20250514", "gpt-4o" 등
    api_key_env: str             # 환경변수 키명
    endpoint: str                # API 엔드포인트
    max_tokens: int = 4096       # 응답 최대 토큰
    timeout_sec: int = 30        # 요청 타임아웃
    cost_per_1k_tokens: float    # 비용 (USD)
    priority: int                # 우선순위 (1=최고)
    capabilities: list[str]      # 지원 역할: ["diagnosis", "recommendation", "validation"]

PROVIDER_CHAIN = [
    LLMProviderConfig(
        name="claude",
        model="claude-sonnet-4-20250514",
        api_key_env="ANTHROPIC_API_KEY",
        endpoint="https://api.anthropic.com/v1/messages",
        cost_per_1k_tokens=0.003,
        priority=1,
        capabilities=["diagnosis", "recommendation", "validation"],
    ),
    LLMProviderConfig(
        name="openai",
        model="gpt-4o",
        api_key_env="OPENAI_API_KEY",
        endpoint="https://api.openai.com/v1/chat/completions",
        cost_per_1k_tokens=0.005,
        priority=2,
        capabilities=["diagnosis", "recommendation", "validation"],
    ),
]
```

#### 3.2.2 프로바이더 선택 우선순위 로직

프로바이더 선택은 **ProviderRouter**가 담당한다. 요청 시점에 가용한 최고 우선순위 프로바이더를 자동 선택하고, 장애 시 fallback 체인을 따른다.

```
요청 발생
    │
    ▼
┌─────────────────────────────────┐
│ 1. Primary 프로바이더 확인       │
│    - API Key 설정 여부          │
│    - Circuit Breaker 상태       │
│    - 월 비용 한도 초과 여부      │
├─────────────────────────────────┤
│  OK → Claude API 호출           │
│  NG ──┐                        │
└───────┼─────────────────────────┘
        │
        ▼
┌─────────────────────────────────┐
│ 2. Fallback 프로바이더 확인      │
│    - 동일한 3가지 체크           │
├─────────────────────────────────┤
│  OK → OpenAI API 호출           │
│  NG ──┐                        │
└───────┼─────────────────────────┘
        │
        ▼
┌─────────────────────────────────┐
│ 3. Degraded Mode (LLM 없이)     │
│    - Optuna 수치 최적화만 수행   │
│    - LLM 진단/해석 부분 스킵     │
│    - 기본 규칙 기반 추천 사용    │
│    - 텔레그램: "LLM 미사용 모드" │
└─────────────────────────────────┘
```

> **비유:** 식당 주방에 수석 셰프(Claude)가 없으면 부셰프(OpenAI)가 대신하고, 둘 다 없으면 레시피북(Optuna 규칙)만으로 요리한다. 음식(파라미터 조정)은 나오지만 셰프의 창의적 판단(맥락 해석)은 빠진다.

#### 3.2.3 Circuit Breaker (프로바이더별 장애 차단)

각 프로바이더에 독립적인 Circuit Breaker를 적용하여 연속 실패 시 자동으로 차단한다.

| 상태 | 조건 | 동작 |
|------|------|------|
| CLOSED (정상) | 실패 < 3회 연속 | 정상 호출 |
| OPEN (차단) | 실패 ≥ 3회 연속 | 즉시 다음 프로바이더로 fallback |
| HALF-OPEN (복구 시도) | OPEN 후 10분 경과 | 1회 시험 호출 → 성공 시 CLOSED, 실패 시 OPEN 유지 |

#### 3.2.4 비용 관리

LLM 호출은 주 1회 튜닝 기준으로 소량이지만, 예상치 못한 재시도나 확장을 대비해 비용 한도를 설정한다.

| 항목 | 기본값 | 설명 |
|------|--------|------|
| `monthly_budget_usd` | 5.0 | 월간 LLM API 비용 한도 |
| `cost_per_tuning_session` | ~0.05 | 1회 튜닝 예상 비용 (입력 ~2K + 출력 ~2K 토큰) |
| `budget_warning_pct` | 0.8 | 한도의 80% 도달 시 텔레그램 경고 |
| `budget_exceeded_action` | "degrade" | 한도 초과 시 Degraded Mode 전환 |

### 3.3 통계 엔진과 LLM의 역할 분담

하이브리드 시스템의 핵심은 "누가 무엇을 하는가"의 명확한 경계다.

```
┌─────────────────────────────────────────────────────────────┐
│                    튜닝 파이프라인 한 사이클                   │
│                                                             │
│  ① 데이터 수집        → 엔진 (DataStore 쿼리)               │
│  ② 지표 계산          → 통계 엔진 (Python 수학 연산)         │
│  ③ 문제 진단          → LLM ★                              │
│  ④ 파라미터 탐색      → 통계 엔진 (Optuna Bayesian Opt.)    │
│  ⑤ 후보 해석/선택     → LLM ★                              │
│  ⑥ 안전 검증          → 통계 엔진 (백테스트 시뮬레이션)      │
│  ⑦ 최종 승인/거부     → LLM ★ (+ Tier 3는 사람 승인)       │
│  ⑧ 반영              → 엔진 (Applier hot-reload)           │
│  ⑨ 변경 사유 기록     → LLM ★ (자연어 요약)                │
│                                                             │
│  ★ = LLM 장애 시 규칙 기반 대체 가능 (Degraded Mode)        │
└─────────────────────────────────────────────────────────────┘
```

#### ③ 문제 진단 — LLM이 하는 일

통계 엔진이 "STR-005 승률 12.5%, PF 0.31"이라는 숫자를 내놓으면, LLM은 이 숫자가 **왜** 나쁜지를 해석한다.

**LLM 진단 프롬프트 예시:**
```
당신은 BTC/KRW 자동매매 전략 분석가입니다.

아래는 STR-005 (Low-Frequency Hybrid) 전략의 지난 7일 성과입니다:
- 레짐: RANGING_LOW_VOL (횡보/저변동)
- 거래 횟수: 8회 (매수 8, 매도 8)
- 승률: 12.5%
- Profit Factor: 0.31
- 평균 보유시간: 2.1시간
- 시그널 정확도: 15%
- 현재 파라미터: buy_threshold=0.12, sell_threshold=-0.12, tf_weights={4h:0.4, 1d:0.6}

다음을 분석해 주세요:
1. 성과 부진의 근본 원인 (1~2가지)
2. 어떤 파라미터를 조정해야 하는지 (Tier 1 파라미터 중)
3. 조정 방향 (올려야/내려야)과 이유

JSON 형식으로 응답하세요:
{
  "root_causes": [...],
  "recommended_params": [{"name": "...", "direction": "increase|decrease", "reason": "..."}],
  "confidence": "high|medium|low"
}
```

**Degraded Mode 대체 (LLM 없을 때):**
```python
# 규칙 기반 진단 - LLM이 없을 때 사용
def rule_based_diagnosis(report: StrategyReport) -> Diagnosis:
    issues = []
    if report.win_rate < 0.30:
        issues.append({"name": "buy_threshold", "direction": "increase",
                        "reason": "win_rate < 30%: 진입 기준 강화 필요"})
    if report.avg_hold_hours < 3.0:
        issues.append({"name": "sell_threshold", "direction": "decrease",
                        "reason": "보유시간 < 3h: 조기 청산 방지"})
    return Diagnosis(causes=issues, confidence="low")  # LLM 대비 낮은 신뢰도
```

> **비유:** 통계 엔진이 혈압·맥박·체온(지표)을 측정하면, LLM 의사가 "이 환자는 감기가 아니라 폐렴 초기"라고 진단한다. 의사가 없으면 매뉴얼(규칙)에 따라 "열이 높으니 해열제"라는 기본 처방만 가능하다.

#### ⑤ 후보 해석/선택 — LLM이 하는 일

Optuna가 상위 3개 파라미터 후보를 내놓으면, LLM이 시장 맥락을 고려하여 최종 1개를 선택한다.

**LLM 선택 프롬프트 예시:**
```
Optuna가 STR-005에 대해 3개 후보를 제안했습니다:

후보 A: buy_threshold=0.144, sell_threshold=-0.10
  → 검증 PF: 1.42, 검증 MDD: 0.8%, 거래 횟수: 3/주

후보 B: buy_threshold=0.16, sell_threshold=-0.09
  → 검증 PF: 1.38, 검증 MDD: 0.5%, 거래 횟수: 2/주

후보 C: buy_threshold=0.13, sell_threshold=-0.11
  → 검증 PF: 1.55, 검증 MDD: 1.2%, 거래 횟수: 5/주

현재 시장 상황:
- BTC/KRW 7일 변동폭: ±1.8%
- 레짐: RANGING_LOW_VOL (12시간 지속)
- 최근 거래소 거래량: 평소 대비 70%

어떤 후보가 가장 적합한지 선택하고 근거를 설명하세요.
```

**Degraded Mode 대체:**
```python
# LLM 없을 때: 검증 PF가 가장 높은 후보를 자동 선택
def rule_based_select(candidates: list[Candidate]) -> Candidate:
    return max(candidates, key=lambda c: c.validation_pf)
```

#### ⑦ 최종 승인/거부 — LLM이 하는 일

안전 검증(백테스트)을 통과한 후보에 대해, LLM이 마지막으로 "이 변경이 정말 합리적인가"를 판단한다. 특히 Tier 2 파라미터(리스크 관련)의 변경에서 중요하다.

**LLM이 거부할 수 있는 케이스:**
- Optuna가 SL을 넓히라고 제안하지만, 최근 시장에 flash crash 패턴이 관측될 때
- 검증 PF는 높지만 특정 날에 수익이 편중되어 있을 때 (운 vs 실력 구분)
- 최근 레짐 전환이 잦아서 안정화 기간이 필요할 때

### 3.5 모듈별 역할 (보강)

**Evaluator (성과 평가기)**

일정 기간(평가 윈도우)의 거래 결과를 수집하고 전략 성과를 계량화한다.

- 입력: 기간 내 모든 체결 주문, 포지션 이력, 시그널 로그
- **통계 엔진 담당** — 정량 지표 계산:
  - 수익률 (총 수익 / 시작 자본)
  - Profit Factor (총 이익 / 총 손실)
  - 최대 낙폭 (MDD)
  - 승률 (Win Rate)
  - 평균 R-배수 (실현 수익 / 초기 리스크)
  - 시그널 정확도 (BUY 시그널 후 실제 상승 비율)
- **LLM 담당** — 정성 진단:
  - 성과 부진 근본 원인 해석
  - 레짐-전략 궁합 평가 ("이 레짐에서 이 전략이 왜 안 맞는지")
  - 조정 우선순위 제안 ("threshold 문제인지, weight 문제인지")
- 출력: `StrategyReport` (정량 요약 + LLM 진단 텍스트)

**Optimizer (하이브리드 탐색기)**

Evaluator의 결과를 기반으로 더 나은 파라미터 조합을 탐색한다.

- **통계 엔진 담당** — 파라미터 수치 탐색:
  1. **Optuna (Bayesian Optimization)** — 소수의 시행으로 최적점 탐색 (기본)
  2. **Grid Search** — 넓은 탐색 시 사용
  3. **유전 알고리즘 (GA)** — 복잡한 다차원 탐색 시 (향후 고려)
- **LLM 담당** — 탐색 결과 해석 및 선택:
  - Optuna 상위 후보 중 시장 맥락에 가장 적합한 1개 선택
  - 후보 간 트레이드오프 설명 (PF vs MDD vs 거래빈도)
  - 최종 승인 판단 + 변경 사유 자연어 작성
- 탐색 시 Walk-Forward 검증 필수: 학습 구간 → 검증 구간 분리
- 파라미터 변경 폭 제한: 1회 조정 시 기존 값 대비 ±20% 이내

**Applier (엔진 반영기)**

탐색된 최적 파라미터를 실제 엔진에 반영한다.

- `SignalGenerator.apply_preset()` 호출로 hot-reload
- 변경 전 현재 파라미터를 `tuning_history` 테이블에 백업
- 변경 사유(LLM 생성), 평가 지표, 이전/이후 값을 모두 기록
- LLM 프로바이더 정보도 기록 (어떤 모델이 판단했는지)

### 4.2 안전장치

AI 튜닝이 아무리 자동화되어도 "자율주행차의 안전벨트"처럼 반드시 필요한 보호 장치가 있다.

**변경 폭 제한 (Guardrails)**

| 규칙 | 설명 |
|------|------|
| 1회 변경 폭 ≤ ±20% | 급격한 파라미터 변화 방지 |
| Tier 2 파라미터: 2회 연속 같은 방향 변경 금지 | SL/TP의 지속적 완화 차단 |
| Tier 3 파라미터: 사람 승인 필수 | 레짐 체계 변경은 수동 확인 |
| MDD 악화 시 자동 롤백 | 변경 후 48시간 내 MDD가 2배 이상 증가하면 이전 값으로 복구 |

**롤백 메커니즘**

```
파라미터 변경 후 48시간 모니터링
    │
    ├─ MDD > 기존 MDD × 2.0 → 즉시 롤백
    ├─ 연속 손실 > max_consecutive_losses → 즉시 롤백
    └─ 48시간 경과 + 이상 없음 → 변경 확정
```

## 5. 예제: 실제 튜닝 시나리오

### 시나리오 A: 횡보장에서 과매매 발생

**상황:** 최근 7일간 RANGING_LOW_VOL 레짐에서 STR-005가 8회 매수 → 7회 손절. 승률 12.5%.

**Evaluator 분석:**
```
STR-005 성과:
  승률: 12.5% (기준: 40% 이상)
  Profit Factor: 0.31 (기준: 1.0 이상)
  시그널 정확도: 15% (BUY 후 1% 이상 상승 비율)
  → 문제: buy_threshold(0.12)가 너무 낮아 노이즈성 시그널에 반응
```

**Optimizer 제안:**
```
현재: buy_threshold = 0.12, sell_threshold = -0.12
제안: buy_threshold = 0.15, sell_threshold = -0.10
근거: walk-forward 결과 PF 0.31 → 1.42 개선, 거래 횟수 8→3 감소
```

**적용 결과:**
```
presets.py 변경:
  STR-005.buy_threshold: 0.12 → 0.15 (+25%, 상한 20% 초과로 0.144로 클램핑)
  STR-005.sell_threshold: -0.12 → -0.10 (기존 대비 17% 변경, 허용 범위 내)
```

### 시나리오 B: 추세장에서 수익 미실현

**상황:** BULL_TREND_HIGH_VOL 레짐에서 STR-002로 매수 후 가격이 5% 상승했으나 trailing stop이 너무 가까워 조기 청산됨.

**Evaluator 분석:**
```
STR-002 성과:
  평균 보유 시간: 4.2시간 (너무 짧음)
  평균 수익률: +0.8% (추세장 대비 미흡)
  trailing_distance_pct가 1.5%로 추세장 변동폭 대비 타이트
```

**Optimizer 제안:**
```
현재: trailing_distance_pct = 0.015, trailing_activation_pct = 0.01
제안: trailing_distance_pct = 0.025, trailing_activation_pct = 0.015
근거: 추세장에서 ATR이 평균 2.5%이므로 트레일링 거리를 ATR 1배 수준으로 확대
```

### 시나리오 C: 레짐 감지 지연

**상황:** 실제로는 추세장이 시작된 지 12시간이 지났지만, debounce(3회) + cooldown(60분) 때문에 레짐 전환이 3.5시간 후에야 완료됨. 초기 상승분을 놓침.

**Evaluator 분석:**
```
레짐 전환 지연:
  실제 추세 시작: 03-20 14:00
  레짐 감지 완료: 03-20 17:30 (3시간 30분 지연)
  이 기간 BTC 상승: +2.8%
  → debounce_count=3 + cooldown=60min이 합쳐져 전환 지연
```

**Optimizer 제안 (Tier 3 — 사람 승인 필요):**
```
현재: debounce_count = 3, cooldown_minutes = 60
제안: debounce_count = 2, cooldown_minutes = 45
⚠️ Tier 3 파라미터 → 텔레그램으로 승인 요청 발송
   "레짐 파라미터 변경 제안: debounce 3→2, cooldown 60→45분. 승인하시겠습니까?"
```

### 6.1 튜닝 이력 테이블 (`tuning_history`)

```sql
CREATE TABLE tuning_history (
    id              INTEGER PRIMARY KEY AUTOINCREMENT,
    tuning_id       TEXT NOT NULL,          -- 튜닝 세션 고유 ID
    created_at      TEXT NOT NULL,          -- 실행 시각
    strategy_id     TEXT NOT NULL,          -- 대상 전략 (STR-001 등)
    tier            INTEGER NOT NULL,       -- 파라미터 계층 (1/2/3)
    parameter_name  TEXT NOT NULL,          -- 변경 파라미터명
    old_value       REAL NOT NULL,          -- 변경 전 값
    new_value       REAL NOT NULL,          -- 변경 후 값
    change_pct      REAL NOT NULL,          -- 변경 비율 (%)
    reason          TEXT NOT NULL,          -- 변경 근거 (LLM 생성 or 규칙 기반)
    eval_window     TEXT NOT NULL,          -- 평가 기간 (예: "2026-03-10~2026-03-17")
    eval_pf         REAL,                   -- 평가 기간 Profit Factor
    eval_mdd        REAL,                   -- 평가 기간 MDD
    eval_winrate    REAL,                   -- 평가 기간 승률
    validation_pf   REAL,                   -- 검증 기간 PF
    validation_mdd  REAL,                   -- 검증 기간 MDD
    llm_provider    TEXT,                   -- 판단에 사용된 LLM ("claude", "openai", "degraded")
    llm_model       TEXT,                   -- 모델명 ("claude-sonnet-4-20250514", "gpt-4o", NULL)
    llm_diagnosis   TEXT,                   -- LLM 진단 원문 (JSON)
    llm_confidence  TEXT,                   -- LLM 자체 신뢰도 ("high", "medium", "low")
    status          TEXT NOT NULL,          -- applied / rejected / rolled_back
    rollback_at     TEXT                    -- 롤백 시각 (있을 경우)
);
```

- `engine/tuner/optimizer.py` — Optuna 통합 + LLM 후보 선택 파이프라인
- `engine/tuner/prompts.py` — 진단/선택/승인 프롬프트 템플릿 관리
- `engine/tuner/degraded.py` — 규칙 기반 대체 로직 (LLM 장애 시)
- Walk-Forward 백테스트 연동 (기존 `run_backtest_all` 스크립트 재활용)
- 파라미터 변경 폭 제한(guardrails) 로직
- LLM 비용 추적 + 월간 한도 관리
- **검증:** Degraded Mode에서도 합리적 파라미터가 선택되는지 A/B 비교

### Phase 3: Applier + 안전장치 (1~2주)

- `engine/tuner/applier.py` — hot-reload + 이력 저장 (LLM 모델명 포함)
- `tuning_history` 테이블 마이그레이션
- 48시간 모니터링 + 자동 롤백 로직
- Tier 3 변경 시 텔레그램 승인 플로우
- **검증:** 롤백 시 LLM 판단 이력과 통계 지표 모두 보존 확인

### Phase 4: 자동화 + 대시보드 (1~2주)

- 스케줄러에 주간 튜닝 태스크 등록
- 대시보드에 튜닝 이력 시각화 추가
- 파라미터 변화 추이 차트 + LLM 진단 요약 표시
- LLM 비용 대시보드 (월간 사용량, 프로바이더별 호출 횟수)
- 롤백 버튼 (수동 제어)
- API 엔드포인트: `GET /tuning/history`, `POST /tuning/rollback/{id}`, `GET /tuning/provider-status`

## 9. 리스크와 대응

**과적합 리스크:** AI가 최근 데이터에만 최적화하여 미래 시장에 부적합한 파라미터를 제안할 수 있다. 이를 방지하기 위해 반드시 Walk-Forward 검증(학습/검증 기간 분리)을 수행하고, 1회 변경 폭을 ±20%로 제한한다.

**연속 롤백 리스크:** 시장이 급변하면 AI가 변경 → 롤백 → 재변경을 반복할 수 있다. 3회 연속 롤백 시 자동 튜닝을 일시 중단하고 수동 개입을 요청한다.

**데이터 부족 리스크:** 거래 빈도가 낮으면(주 3건 미만) 통계적으로 유의미한 평가가 불가능하다. 최소 거래 건수 미달 시 튜닝을 스킵하고 "데이터 부족" 상태를 보고한다.

**LLM 프로바이더 장애 리스크:** 외부 API에 의존하므로 장애, 지연, 모델 변경 등이 발생할 수 있다. ProviderRouter의 Circuit Breaker + 3단계 fallback(Primary → Fallback → Degraded)으로 대응하며, Degraded Mode에서도 Optuna 기반 수치 최적화는 정상 동작한다. 다만 Degraded Mode가 2주 이상 지속되면 수동 개입 알림을 발송한다.

**LLM 할루시네이션 리스크:** LLM이 시장 데이터를 잘못 해석하거나 존재하지 않는 패턴을 "발견"할 수 있다. 이를 방지하기 위해 LLM의 판단은 항상 Optuna의 통계 검증을 거친 후보에 대해서만 적용하며, LLM이 단독으로 파라미터 값을 생성하지 않는다. LLM은 "선택"만 하고, "생성"은 통계 엔진이 한다.

**프로바이더 간 일관성 리스크:** Claude와 OpenAI가 동일한 데이터에 다른 판단을 내릴 수 있다. 프로바이더 전환 시에는 이전 프로바이더의 마지막 판단도 함께 기록하고, 연속성 있는 튜닝을 위해 가능한 한 같은 프로바이더를 유지한다. 세션 중간에 프로바이더가 바뀌면 텔레그램으로 알림한다.

**비용 폭주 리스크:** LLM API 호출 비용이 예상을 초과할 수 있다. 월 $5 한도 + 80% 경고 + 초과 시 자동 Degraded 전환으로 대응한다. 주 1회 튜닝 기준 1회당 약 $0.05로 월 $0.20~$0.25 수준이므로 정상 운영 시 문제없으나, 재시도/디버깅 시 증가할 수 있다.
