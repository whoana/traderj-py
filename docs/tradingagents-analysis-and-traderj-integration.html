Part 1. TradingAgents 레포 분석

1.1 프로젝트 개요

TradingAgents는 LLM 기반 멀티 에이전트 프레임워크로, 실제 트레이딩 팀의 역할 분담 구조를 시뮬레이션합니다. 비유하자면, "AI 투자 회사"를 통째로 만들어 놓은 것 — 애널리스트, 리서처, 트레이더, 리스크 매니저가 각자 역할을 수행하고 토론을 통해 의사결정을 내립니다.

1.3 핵심 메커니즘 상세

Structured Debate (구조적 토론)

Bull/Bear 리서처가 N라운드 토론 후 Investment Judge가 종합 판단합니다. 비유하면 법정 변론과 같습니다 — 검찰(Bear)과 변호인(Bull)이 각자 논거를 펴고 판사(Judge)가 판결합니다.

  • Bull: 상승 근거 + 기회 요인 제시
  • Bear: 하락 위험 + 리스크 요인 반박
  • Judge: 양측 논거 종합 → 최종 방향 결정
  • max_debate_rounds로 토론 깊이 조절

Risk Debate (리스크 토론)

리스크 관리에도 3인 토론 구조를 적용합니다:

  • Aggressive Debator: "수익 기회가 크니 리스크를 감수하자"
  • Conservative Debator: "변동성이 높으니 축소/거부하자"
  • Neutral Debator: 양측 의견 중재, 균형점 제안

Reflection & Memory (회고 학습)

비유하면 트레이딩 일지를 자동으로 쓰는 것. 매 사이클 후 각 에이전트가 "이번에 뭘 잘했고 뭘 틀렸는지" 반성합니다.

  • Bull/Bear/Trader/Judge/RiskMgr 각각 독립 회고
  • 시장 상황 + 결과(손익)를 묶어 메모리에 저장
  • 다음 사이클에서 유사 상황 조회 → 판단에 반영
  • memory.add_situations([(situation, result)])

Signal Processing (시그널 처리)

Trader Agent의 분석 결과에서 최종 매매 시그널을 추출합니다. LLM의 자연어 출력에서 BUY/SELL/HOLD + 확신도 + 포지션 사이즈를 구조화합니다.

핵심 차이점 요약

TradingAgents는 "AI가 토론해서 판단"하는 모델이고, TraderJ는 "수학 공식으로 계산해서 판단"하는 모델입니다. 비유하면 TradingAgents는 투자위원회 회의, TraderJ는 자동 계산기에 가깝습니다.

두 접근 방식은 상호 배타적이지 않습니다. TraderJ의 빠르고 결정론적인 엔진 위에 TradingAgents의 핵심 아이디어를 선별적으로 얹는 것이 최적의 전략입니다.

Part 3. TraderJ에 반영 가능한 사항 (7개)

1. Sentiment Analyst — 감성 분석 레이어 추가

P0 - 최우선

TradingAgents 방식: LLM이 소셜미디어/뉴스 감성을 자연어로 분석하여 점수화

TraderJ 현재: Fear & Greed Index, 김치프리미엄 등 매크로 점수만 반영 (macro_weight)

반영 방안: LLM 감성분석 모듈 추가

# 새로운 컴포넌트: engine/strategy/sentiment_analyzer.py

class SentimentAnalyzer:
    """
    주기적으로(예: 4시간) LLM에 최근 뉴스/소셜 데이터를 입력하여
    감성 점수(-1.0 ~ +1.0)를 산출.
    SignalGenerator의 macro_score 자리에 통합.
    """

    async def analyze(self, symbol: str) -> SentimentResult:
        # 1. 뉴스 데이터 수집 (CryptoPanic, Reddit, X API)
        news = await self.fetch_recent_news(symbol)

        # 2. LLM에 감성 분석 요청
        prompt = f"""
        다음 {symbol} 관련 뉴스를 분석하고 감성 점수를 산출하세요.
        점수: -1.0(극도 부정) ~ +1.0(극도 긍정)
        뉴스: {news}
        """
        result = await self.llm_client.analyze(prompt)

        # 3. 구조화된 결과 반환
        return SentimentResult(
            score=result.score,        # -1.0 ~ +1.0
            confidence=result.conf,     # 0.0 ~ 1.0
            key_factors=result.factors,
            timestamp=now()
        )

시그널 파이프라인 통합 위치

# 현재 8단계 파이프라인의 Step 7을 확장:
# 기존: technical × (1 - macro_weight) + macro × macro_weight
# 변경: technical × tech_w + macro × macro_w + sentiment × sent_w
#       (tech_w + macro_w + sent_w = 1.0)

final_score = (
    technical_score * preset.tech_weight +      # 예: 0.65
    macro_score * preset.macro_weight +          # 예: 0.15
    sentiment_score * preset.sentiment_weight    # 예: 0.20  ← 신규
)

기술적 지표가 놓치는 "시장 분위기" 포착
급변 뉴스(규제, 해킹 등)에 빠른 반응
매크로 점수보다 세분화된 분석		 LLM API 비용 (4시간 주기로 제한)
감성 점수의 지연 (실시간 아님)
환각(hallucination) 리스크 → confidence 임계값 필요

구현 복잡도: 중 예상 기간: 1~2주

2. Bull/Bear Debate — 매매 시그널 교차 검증

P0 - 최우선

TradingAgents 방식: Bull/Bear 리서처가 N라운드 토론 후 Judge가 종합 판단

TraderJ 현재: 단일 시그널 점수 → 임계값 비교 → 즉시 BUY/SELL/HOLD

반영 방안: LLM 기반 시그널 2차 검증기

비유하면, 현재 TraderJ는 "계산기가 OK하면 바로 매수"하는데, 여기에 "투자위원회 승인" 단계를 추가하는 것입니다.

# engine/strategy/signal_validator.py

class SignalValidator:
    """
    SignalGenerator가 BUY/SELL 시그널을 생성하면,
    LLM에 Bull/Bear 관점으로 교차 검증을 요청.
    높은 확신도일 때만 시그널 통과.
    """

    async def validate(self, signal: SignalResult, market_context: dict) -> ValidationResult:
        # 1. Bull 관점 분석
        bull_case = await self.llm_client.analyze(
            f"현재 {signal.direction} 시그널의 강점과 상승 근거를 분석하세요.\n"
            f"기술지표: {market_context['indicators']}\n"
            f"시장상황: {market_context['regime']}"
        )

        # 2. Bear 관점 분석
        bear_case = await self.llm_client.analyze(
            f"현재 {signal.direction} 시그널의 약점과 하락 위험을 분석하세요.\n"
            f"기술지표: {market_context['indicators']}\n"
            f"시장상황: {market_context['regime']}"
        )

        # 3. 종합 판단 (Judge)
        judgment = await self.llm_client.analyze(
            f"Bull 의견: {bull_case}\nBear 의견: {bear_case}\n"
            f"원본 시그널: {signal.direction} (점수: {signal.score})\n"
            f"이 시그널을 승인하시겠습니까? (approve/reject/reduce_size)"
        )

        return ValidationResult(
            approved=judgment.decision,
            adjusted_score=judgment.adjusted_score,
            reasoning=judgment.reasoning
        )

# engine/loop/trading_loop.py 수정
# 기존: signal → risk_check → execute
# 변경: signal → LLM_validate → risk_check → execute

async def _process_signal(self, signal: SignalResult):
    if signal.direction in (Direction.BUY, Direction.SELL):
        # 새로운 2차 검증 단계
        validation = await self.signal_validator.validate(signal, self.market_context)

        if not validation.approved:
            logger.info(f"LLM Judge rejected signal: {validation.reasoning}")
            return  # 시그널 거부

        signal = signal.with_adjusted_score(validation.adjusted_score)

    # 기존 리스크 체크 → 주문 실행
    await self.risk_manager.pre_validate(signal)
    await self.order_manager.handle(signal)

3. Reflection Memory — 트레이딩 회고 학습 시스템

P1 - 높음

TradingAgents 방식: 매 사이클 후 LLM이 결과를 반성하고 메모리에 저장, 유사 상황에서 참조

TraderJ 현재: 학습 메커니즘 없음 — 같은 실수를 반복할 수 있음

반영 방안: 트레이드 회고 모듈

비유하면, 트레이딩 일지를 자동으로 작성하고, 다음에 비슷한 상황이 오면 꺼내 읽는 것입니다.

# engine/strategy/reflection.py

class TradeReflector:
    """
    포지션 청산 시 LLM이 해당 트레이드를 분석하고
    교훈을 벡터DB/JSON에 저장.
    다음 BUY 시그널 시 유사 상황 검색 → 판단에 반영.
    """

    async def reflect_on_close(self, position: Position, market_snapshot: dict):
        prompt = f"""
        트레이드 결과를 분석하세요:
        - 진입: {position.entry_price} / 청산: {position.current_price}
        - 손익: {position.realized_pnl} ({position.pnl_pct}%)
        - 보유기간: {position.holding_duration}
        - 진입 시 레짐: {position.entry_regime}
        - 청산 사유: {position.close_reason}

        시장 상황: {market_snapshot}

        질문:
        1. 이 트레이드에서 잘한 점은?
        2. 개선할 점은?
        3. 유사 상황 재발 시 어떻게 해야 하는가?
        """

        reflection = await self.llm_client.analyze(prompt)

        # 메모리에 저장 (상황 → 결과 → 교훈)
        await self.memory_store.save({
            "market_situation": market_snapshot,
            "trade_result": position.to_dict(),
            "lesson": reflection.lesson,
            "regime": position.entry_regime,
            "timestamp": now()
        })

    async def recall_similar(self, current_situation: dict, top_k: int = 3):
        """유사 과거 상황 검색 → 교훈 반환"""
        return await self.memory_store.search_similar(current_situation, top_k)

활용 흐름

포지션 청산 (PositionClosedEvent) │ ▼ TradeReflector.reflect_on_close() │ ├─ LLM이 트레이드 분석 ├─ 교훈 추출 └─ 메모리에 저장 ... 시간 경과 ... 다음 BUY 시그널 발생 │ ▼ TradeReflector.recall_similar(current_market) │ ├─ 유사 과거 상황 3개 검색 ├─ "지난번 이런 상황에서 손실봤음, RSI 과매수 구간 진입은 피할 것" └─ SignalValidator에 컨텍스트로 전달 → 판단에 반영

구현 복잡도: 높음 예상 기간: 2~3주

4. Risk Debate — 리스크 관리 다관점 평가

P1 - 높음

TradingAgents 방식: Aggressive / Conservative / Neutral 3인이 리스크 토론

TraderJ 현재: 단일 RiskManager가 규칙 기반 체크 (ATR, 일일손실, 변동성캡)

반영 방안: 리스크 등급별 포지션 사이즈 조정

전체 구조를 바꾸지 않고, 기존 RiskManager의 pre_validate() 결과에 LLM 기반 2차 리스크 평가를 추가합니다.

# engine/execution/risk_evaluator.py

class RiskEvaluator:
    """
    기존 RiskManager가 PASS한 주문에 대해
    3가지 관점으로 리스크를 재평가하고 사이즈를 조정.
    """

    async def evaluate(self, order: OrderRequest, context: dict) -> RiskAdjustment:
        perspectives = await asyncio.gather(
            self._aggressive_view(order, context),   # "기회 비용" 관점
            self._conservative_view(order, context),  # "최대 손실" 관점
            self._neutral_view(order, context),       # "균형" 관점
        )

        # 3개 관점의 권장 사이즈 비율 중위값 사용
        size_ratios = [p.recommended_size_ratio for p in perspectives]
        adjusted_ratio = sorted(size_ratios)[1]  # 중위값

        return RiskAdjustment(
            original_size=order.amount,
            adjusted_size=order.amount * adjusted_ratio,
            perspectives=perspectives,
            reasoning=self._summarize(perspectives)
        )
관점성향예시 판단
Aggressive수익 극대화"추세 강하니 100% 사이즈 유지"
Conservative손실 최소화"변동성 상승 중, 50%로 축소"
Neutral균형"75% 사이즈 + 타이트한 SL"

class OnchainScorer:
    """
    Glassnode/CryptoQuant 등에서 온체인 데이터를 수집하여
    펀더멘탈 점수를 산출.
    """

    METRICS = {
        "exchange_netflow":    -1,  # 거래소 유출 → 상승 (보유 의지)
        "whale_accumulation":  +1,  # 고래 축적 → 상승
        "miner_selling":       -1,  # 채굴자 매도 → 하락 압력
        "active_addresses":    +1,  # 활성 주소 증가 → 건전한 네트워크
        "nvt_ratio":           -1,  # 높으면 과대평가
        "sopr":                +1,  # >1 이면 수익실현, <1 이면 손절 (반전 신호)
    }

    async def score(self) -> float:
        """종합 온체인 점수 (-1.0 ~ +1.0)"""
        raw_data = await self.fetch_onchain_data()
        weighted_sum = sum(
            self.normalize(raw_data[metric]) * weight
            for metric, weight in self.METRICS.items()
        )
        return tanh(weighted_sum)  # -1 ~ +1 클램핑

async def handle_regime_transition(self, old_regime, new_regime, position):
    if position is None:
        # Case C: 포지션 없음 → 깔끔한 전환
        return SwitchAction.CLEAN_SWITCH

    # 기존 규칙 기반 판단
    unrealized_pnl_pct = position.unrealized_pnl_pct

    if unrealized_pnl_pct >= 0:
        # 수익 중 → 즉시 청산 (이익 확보)
        return SwitchAction.CLOSE_AND_SWITCH

    elif unrealized_pnl_pct > -2.0:
        # 소폭 손실 → 즉시 청산 (손실 제한)
        return SwitchAction.CLOSE_AND_SWITCH

    else:
        # 큰 손실 중 → LLM에게 판단 위임
        judgment = await self.llm_judge.evaluate(
            f"""
            레짐 전환: {old_regime} → {new_regime}
            기존 포지션 미실현 손실: {unrealized_pnl_pct}%

            선택지:
            A) 즉시 시장가 청산 (손실 확정, 새 전략 시작)
            B) SL을 타이트하게 조이고 대기 (반등 기회 + 제한된 추가 손실)
            C) 현재 포지션 유지, 새 전략 진입 보류

            현재 시장 상황과 새 레짐을 고려하여 최적 선택은?
            """
        )
        return judgment.action

소폭 손실 (-2% ~ 0%)즉시 청산 + 전환손실 제한, 빠른 전환
